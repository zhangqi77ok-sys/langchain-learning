# 03-multi-node-workflow

这一组只讲一件事：
**把一个流程拆成多个职责清晰的节点。**

前两组你已经学到：

- 最小直线图怎么跑
- 条件分支怎么选路

这一步继续往前走：

**不是只会分流，而是会把一个任务拆成多个可维护的处理阶段。**

## 你要学会什么

- 多节点串联怎么写
- 每个节点只做一件事是什么意思
- 为什么中间状态比“大杂烩函数”更适合工作流
- 为什么最后一个节点通常负责收口

## 这个案例做了什么

这个案例会把一个简单的问答流程拆成 4 个节点：

1. `normalize_question`
2. `draft_answer`
3. `polish_answer`
4. `finalize_answer`

流程是：

`START -> normalize_question -> draft_answer -> polish_answer -> finalize_answer -> END`

每个节点只负责自己的那一小段工作。

## 这一组最重要的认知

### 1. 节点越清晰，流程越容易维护

如果一个节点同时做很多事：

- 清洗输入
- 生成答案
- 修饰语言
- 检查格式

那后面一改逻辑就容易牵一发动全身。

LangGraph 的价值之一，就是逼你把职责拆开。

### 2. 中间状态就是节点之间的接口

节点和节点之间不是直接互相调用，
而是靠状态传递。

这意味着：

- 前一个节点只要把结果写对
- 后一个节点就能从状态里拿到它需要的内容

这种方式比“层层传参”更适合复杂流程。

### 3. 最后一节点负责收口

工作流到最后，一般会做两件事：

- 把中间结果整理成最终输出
- 只暴露真正需要给外部看的字段

这会让图的出口更清楚。

## 运行方式

在仓库根目录 `E:\langchain-learning` 下执行：

```powershell
.\.venv\Scripts\python advanced\langgraph\03-multi-node-workflow\demo.py
```

## 你会看到什么

你会看到每个节点依次打印：

- 它读到了什么
- 它写回了什么
- 状态是如何一步步变完整的

## 这在真实项目里有什么价值

这一组最接近真实工作流。

因为真实项目里常见的流程往往就是：

- 输入规范化
- 意图判断
- 检索或工具调用
- 答案生成
- 答案整理

如果你能把这四步想清楚，
后面无论是做 RAG、Agent 还是 Web 工作流，都会稳很多。

## 建议你自己改的地方

1. 把 `user_question` 改成更长的问题
2. 给 `draft_answer` 增加一个固定模板
3. 在 `polish_answer` 里加入更明显的“润色”逻辑

## 这一组的目标

让你真正形成一个习惯：

**复杂任务不是堆到一个函数里，而是拆成多个小节点。**
