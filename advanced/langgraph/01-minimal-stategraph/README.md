# 01-minimal-stategraph

这一组只讲一件事：
**LangGraph 最小图到底是怎么跑起来的。**

前面的 LangChain 基础更偏向：

- 调模型
- 组 prompt
- 做 RAG
- 调工具

而 LangGraph 更偏向：

- 把一个任务拆成多个节点
- 规定节点之间怎么流转
- 让状态在节点之间传递

所以这第一组先不接大模型，只看最小骨架。

## 你要学会什么

- 什么是 `State`
- 什么是节点 `node`
- 什么是边 `edge`
- `START` 和 `END` 的作用
- 一个最小两节点图如何执行

## 这个案例做了什么

这个案例构建了一个只有两个节点的图：

1. `prepare_topic`
2. `build_summary`

执行顺序是：

`START -> prepare_topic -> build_summary -> END`

图在运行时会维护一个共享状态对象。
每个节点拿到当前状态后，返回自己要更新的字段。
LangGraph 会把这些更新合并回状态，再流转到下一个节点。

## 这组代码里最重要的认知

### 1. 节点不是随便写的普通函数

在 LangGraph 里，节点函数通常要遵守一个很核心的约定：

- 输入：当前状态
- 输出：对状态的更新

也就是说，节点的本质不是“打印点什么”，而是“消费状态并产生新的状态变化”。

### 2. 边决定顺序，节点决定处理逻辑

- 节点负责“做什么”
- 边负责“下一步去哪里”

这和真实项目很像：
你会把“查资料”“生成答案”“审核答案”拆成不同节点，
再用边把它们串成一个工作流。

### 3. `StateGraph` 是工作流骨架

如果把 LangChain 看成能力组件层，
那 `StateGraph` 可以先理解成流程编排层。

它最直接的价值是：
**把多步任务从一坨代码，变成结构清晰、可扩展的图。**

## 运行方式

在仓库根目录 `E:\langchain-learning` 下执行：

```powershell
.\.venv\Scripts\python advanced\langgraph\01-minimal-stategraph\demo.py
```

## 你会看到什么

你会看到：

- 初始状态
- 第一个节点如何写入 `topic`
- 第二个节点如何基于 `topic` 生成 `summary`
- 最终完整状态

## 这在真实项目里有什么价值

这组本身很简单，但价值非常实际。

真实项目里常见的流程，例如：

- 用户问题分类
- 决定走普通回答还是 RAG
- 检索资料
- 生成回答
- 做结果检查

本质上都可以拆成“节点 + 边 + 状态”。

所以你现在学的不是一个玩具 API，
而是在学后面复杂 Agent 工作流的骨架。

## 建议你自己改的地方

1. 把初始 `user_input` 改成别的话题
2. 在状态里再加一个字段，比如 `difficulty`
3. 再加一个第三节点，让它对 `summary` 做一次改写

## 这一组的目标

让你先把这句话真正吃透：

**LangGraph 的第一核心，不是模型，而是状态驱动的流程编排。**
