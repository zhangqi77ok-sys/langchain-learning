# 02-conditional-branching

这一组只讲一件事：
**LangGraph 里怎么根据状态走不同分支。**

上一组你已经看到最小直线流程：

`START -> 节点1 -> 节点2 -> END`

但真实项目里，流程通常不是一条直线。
更常见的是：

- 简单问题，直接回答
- 复杂问题，先检索资料再回答
- 命中工具需求，先调工具

所以这一组要学的是：
**图不是只能顺着跑，它还能根据中间结果决定下一步。**

## 你要学会什么

- 什么是条件分支
- 什么是路由函数
- 条件边和普通边的区别
- 为什么“分支判断”和“实际处理”要拆开

## 这个案例做了什么

这个案例会构建一个最小分支图。

流程大致是：

1. 先进入 `classify_question`
2. 根据问题内容判断它是 `simple` 还是 `complex`
3. 如果是 `simple`，走 `answer_simple_question`
4. 如果是 `complex`，走 `suggest_rag_path`
5. 最后结束

也就是：

- `START -> classify_question`
- `classify_question -> simple 分支`
- `classify_question -> complex 分支`
- 分支节点 -> `END`

## 这一组最重要的认知

### 1. 节点负责处理，路由函数负责选路

这是 LangGraph 很重要的一个拆分：

- 节点：真正做事，读状态、写状态
- 路由函数：不负责干活，只负责决定下一步去哪里

这个拆分很有价值。
因为真实项目里，判断逻辑和处理逻辑混在一起时，代码会很快变乱。

### 2. 条件分支的本质是“先分类，再流转”

很多 AI 工作流本质上都符合这个模式：

- 先判断用户意图
- 再进入对应处理链路

比如：

- 闲聊 -> 普通回答
- 知识问答 -> RAG
- 查天气 -> 工具调用

你现在学的这个最小案例，就是后面大工作流的骨架缩小版。

### 3. 分支不是为了炫技，而是为了可控

如果所有逻辑都写进一个大节点里，
你虽然也能跑通，
但后面想增加规则、测试判断逻辑、替换某个分支都会很麻烦。

图的分支结构能让流程更清楚，也更容易扩展。

## 运行方式

在仓库根目录 `E:\langchain-learning` 下执行：

```powershell
.\.venv\Scripts\python advanced\langgraph\02-conditional-branching\demo.py
```

## 你会看到什么

你会看到两次运行结果：

1. 一个简单问题
2. 一个复杂问题

程序会打印：

- 分类节点如何写入 `question_type`
- 路由函数如何决定下一步
- 不同分支节点如何写入不同答案
- 最终状态有什么区别

## 这在真实项目里有什么价值

这组的实际价值非常大。

因为后面你做这些内容时，几乎都会用到分支：

- 多知识库路由
- 普通回答和 RAG 切换
- 工具调用前置判断
- 是否需要人工审核
- 是否需要重试或兜底

所以“条件分支”不是边缘能力，
而是 LangGraph 进入项目级工作流的第一道门。

## 建议你自己改的地方

1. 把复杂问题判断规则改得更严格一点
2. 再增加一个分支，比如 `tool`
3. 把最终答案字段改名为 `response`，观察代码会改哪些地方

## 这一组的目标

让你真正理解：

**LangGraph 不是把步骤串起来而已，它更重要的是把决策过程显式化。**
