# 01-bm25-vector-fusion

这一组只讲一件事：
**BM25 和向量检索怎么合起来用。**

单独用 BM25 的问题是：

- 依赖关键词命中
- 同义表达容易漏掉

单独用向量检索的问题是：

- 更偏语义
- 但有时对精确关键词不如 BM25 稳

所以混合检索的核心价值是：
**让词面匹配和语义匹配互补。**

## 你要学会什么

- BM25 在做什么
- 向量检索在做什么
- 为什么要把两路结果融合
- 为什么融合后的排序通常比单一路径更稳

## 这个案例做了什么

这个案例会用一小组本地文本做检索，流程是：

1. 准备文档集
2. 用 BM25 召回一批候选
3. 用向量相似度召回一批候选
4. 把两路结果归一化后融合打分
5. 输出最终 top-K

它不会接 LLM 生成回答。
这一版只关注检索层本身。

## 这一组最重要的认知

### 1. BM25 擅长“字面相关”

如果用户问的词，正好在文档里出现得很明显，BM25 往往很有用。

它适合：

- 关键词明确的问题
- 专有名词搜索
- 精确短语匹配

### 2. 向量检索擅长“语义相关”

如果用户说法和文档不完全一样，但意思接近，向量检索更容易找出来。

它适合：

- 同义改写
- 口语表达
- 概念相近但措辞不同的问题

### 3. 混合检索的本质是互补

很多真实问题既需要：

- 看关键词
- 也看语义

所以混合方案通常更稳。

## 运行方式

在仓库根目录 `E:\langchain-learning` 下执行：

```powershell
.\.venv\Scripts\python advanced\hybrid-retrieval\01-bm25-vector-fusion\demo.py
```

## 你会看到什么

你会看到：

- BM25 的 top-K
- 向量检索的 top-K
- 融合后的最终排序
- 每篇文档的最终分数

## 这在真实项目里有什么价值

这就是很多 RAG 系统的基础检索层。

尤其是：

- 知识库问答
- 技术文档搜索
- 多知识库检索
- 企业内部检索

实际都经常会用“BM25 + 向量”的组合，而不是只押一边。

## 建议你自己改的地方

1. 改查询词，看看三种排序怎么变化
2. 增加一批同义表达文档
3. 把 `top_k` 改小或改大，观察候选变化

## 这一组的目标

让你先把这句话吃透：

**混合检索不是重复检索，而是让不同检索方式互相补短。**
