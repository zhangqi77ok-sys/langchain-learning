LangChain 是一个帮助开发者构建大语言模型应用的开发框架。

它的目标不是替代大模型，而是把大模型应用开发中常见的能力组织起来，例如提示词管理、模型调用、输出解析、文档处理、检索增强生成、工具调用和多步流程编排。

当开发者只需要做一个简单问答时，直接调用模型 API 往往已经足够。
但当系统开始需要知识库问答、多轮对话、结构化输出、外部工具接入或复杂工作流时，代码复杂度会明显上升。

LangChain 的价值在于提供一套相对统一的抽象，让这些环节可以更容易组合。

在 RAG 场景中，常见流程是：
先准备文档；
再把文档切分成较小的文本块；
然后用 embedding 模型把文本块转换成向量；
用户提问时，再把问题也转换成向量；
系统通过相似度检索找到最相关的文本块；
最后把这些文本块作为上下文交给聊天模型生成答案。

Tool Calling 的核心思想是：让模型在回答前，先判断是否需要借助外部工具。
例如天气查询、数据库查询、搜索接口、代码执行器，都可以被建模成工具。

Agent 可以看成是建立在工具调用之上的更高一层抽象。
它不是只调用一次工具，而是允许模型根据中间结果继续决策，决定是否还要调用别的工具，直到任务结束。

Memory 解决的是多轮对话中的上下文延续问题。
严格来说，大模型本身并不会永久记住之前的对话。
所谓“记忆”，本质上通常是框架把历史消息重新放回提示词里。

Streaming 则解决了输出体验问题。
如果系统等模型生成完整答案后再统一显示，用户会感觉更慢。
流式输出可以让内容边生成边展示。
